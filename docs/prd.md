# Product Requirements Document - 内部运营中台客户信息管理与运营系统

**Author:** Sacrtap
**Date:** 2026-02-25

## Executive Summary

### 愿景与目标

**内部运营中台客户信息管理与运营系统**是一个企业内部 Web 应用，旨在解决客户数据分散、运营效率低下、客户流失风险高的核心问题。

**核心问题：**
- 1320 个客户数据分散在 Excel 中，缺乏统一管理
- 运营团队每月花费 3 天手工生成结算单，数据不准确
- 193 个僵尸账号（14.6%）未被识别，客户流失风险高
- 93.6% 的客户未进行价值评级，无法精细化运营

**目标用户：**
- **运营经理**：配置定级规则、查看数据分析、管理客户分配
- **运营专员**：日常客户数据维护、健康度监控、客户转移操作
- **销售团队**：查看自己负责的客户信息、了解客户价值等级

**未来愿景：**
打造一个数据驱动的客户运营中台，实现：
- 客户数据集中管理，告别 Excel 分散模式
- 客户健康度自动监控，主动发现流失风险
- 客户价值科学分层，支持精细化运营
- 最终提升运营效率、降低客户流失、增加收入

### What Makes This Special（差异化价值）

**行业定制化：**
- 专为房产行业客户管理设计，贴合业务逻辑
- 基于 1320 条真实客户数据洞察设计功能
- 支持复杂业务场景（一客一价、多层级定级）

**灵活配置化：**
- 客户定级阈值可配置（S/A/B/C/D 五级）
- 支持定价规则灵活调整（第二期实现一客一价）
- 无需硬编码，运营人员可自行调整规则

**数据驱动决策：**
- 僵尸账号自动识别（193 个潜在流失客户）
- 客户价值分层（年消费规模 S/A/B/C/D）
- 客户转移历史记录，可追溯可审计

**分阶段实施策略：**
- 第一期（MVP）：聚焦核心痛点（MDM、健康度监控、价值评估）
- 第二期：扩展高级功能（价格中心、用量采集、结算自动化）
- 快速见效，降低风险，持续迭代

**核心洞察：**
客户数据分散 + 缺乏健康度监控 → 运营效率低下 + 客户流失风险 → 业绩增长受阻

**技术架构：**

- 前端：Vue 3 + Arco Design + TypeScript
- 数据库：PostgreSQL 18（UTF-8）
- 部署：Docker + Docker Compose

**关键业务规则：**
- 客户定级：年消费 ≥100 万（S）、≥50 万（A）、≥20 万（B）、≥10 万（C）、≥1 万（D）
- 定级维护：人工维护，配置文件支持阈值调整
- 客户分配：运营录入时指派销售人员
- 客户转移：运营直接操作，无需审批
- **设备类型**：X/N/L 三个系列，每个系列分单层/多层
- **客户设备**：一个客户可以使用多种设备类型
- **定价模式**：三种模式（定价结算/阶梯结算/包年结算），按客户配置，可随时切换（无需审批）
- **定价结算**：按设备类型×楼层类型配置单价（6 种单价：X 单/X 多/N 单/N 多/L 单/L 多）
- **阶梯结算**：支持按总量或按设备类型分别计算，自定义阶梯，区分单层/多层单价
- **包年结算**：A/B/C/D 四套餐，配置套餐等级和用量，超量按定价结算模式计费

**MVP 范围（第一期，3.5 个月）：**

2. 客户健康度监控（僵尸账号识别）
3. 客户价值评估（人工定级）
4. 基础系统能力（认证/权限/日志/备份）

**第二期功能：**
- 用量数据采集（每日外部接口，区分设备类型×楼层类型）

## E2E 测试开发规则（基于实际开发经验）

> 以下规则基于实际 E2E 测试开发中的问题总结，必须严格遵守以避免重复错误。

### Playwright 选择器规则

**1. 选择器必须精确唯一**
- **问题**: 使用模糊文本选择器（`button:has-text("登录")`）匹配到页面中 3 个按钮
- **根本原因**: 缺乏对页面 DOM 结构的了解，选择器不够精确
- **规则**: E2E 测试中，所有选择器必须唯一标识目标元素
- **实践**: 
  - ✅ 使用 CSS 类名：`.submit-button`（自定义类）
  - ✅ 使用 aria 属性：`[aria-label="登录"]`
  - ✅ 使用 data 属性：`[data-testid="login-button"]`
  - ❌ 避免模糊文本选择，除非确认唯一性

**2. Arco Design 错误消息类名需区分场景**
- **问题**: 表单验证错误和全局 API 错误使用相同的 `.arco-message-error` 类名
- **根本原因**: 不了解 Arco Design 组件库的不同错误展示机制
- **规则**: 根据 UI 组件库的文档，使用正确的错误消息类名
- **实践**:
  - 表单验证：`.arco-form-item-message`（字段级错误）
  - 全局消息：`.arco-message` / `.arco-message-error`（页面级消息提示）
  - 错误状态：`.arco-form-item-has-error`（字段错误状态）

### 组件交互规则

**3. 图标导入必须验证存在性**
- **问题**: 使用 `IconExchange`、`IconLightbulb`、`IconUserSlash` 等 Arco Design 不存在的图标
- **根本原因**: 未查阅 Arco Design 图标文档就直接使用
- **规则**: 引用任何 UI 组件库图标前，必须验证该图标存在
- **实践**:
  - 访问 Arco Design 官方文档查阅可用图标列表
  - 使用 IDE 自动补全功能，避免拼写错误
  - 替代方案：`IconExchange` → `IconSync`，`IconLightbulb` → `IconCompass`

**4. Mock API 路径必须完整匹配**
- **问题**: 测试中使用 `**/auth/login` mock 路由，但实际 API 路径是 `**/api/v1/auth/login`
- **根本原因**: Mock 路由的 glob 模式不够精确，导致实际请求未被拦截
- **规则**: Mock API 路径必须与后端实际 API 路径完全一致
- **实践**:
  ```typescript
  // ❌ 错误：路径不完整
  await page.route('**/auth/login', ...)
  
  // ✅ 正确：完整路径
  await page.route('**/api/v1/auth/login', ...)
  ```

**5. Pinia Store 异步函数应返回值**
- **问题**: `userStore.login()` 函数没有返回 API 响应数据，导致 `await userStore.login()` 后代码无法继续执行
- **根本原因**: 异步函数设计不完整，未考虑调用方的使用场景
- **规则**: Store 中的异步操作函数应返回操作结果
- **实践**:
  ```typescript
  // ❌ 错误：无返回值
  const login = async (username, password) => {
    const response = await fetch(...)
    // 处理响应，但不返回
  }
  
  // ✅ 正确：返回响应数据
  const login = async (username, password) => {
    const response = await fetch(...)
    const data = await response.json()
    // 处理响应
    return data  // 返回给调用方
  }
  ```

**6. Arco Design Checkbox 使用隐藏 input**
- **问题**: Checkbox 的 `<input type="checkbox">` 元素通过 CSS 隐藏，`toBeVisible()` 断言失败
- **根本原因**: 不了解 Arco Design Checkbox 的实现方式（隐藏原生 input，使用自定义样式）
- **规则**: 对于自定义 UI 组件，验证方式需要根据组件实现调整
- **实践**:
  ```typescript
  // ❌ 错误：直接验证 input
  await expect(page.locator('input[type="checkbox"]')).toBeVisible()
  
  // ✅ 正确：验证元素存在或父容器
  await expect(page.locator('input[type="checkbox"]')).toHaveCount(1)
  // 或验证自定义容器
  await expect(page.locator('.arco-checkbox')).toBeVisible()
  ```

### 测试数据规则

**7. 表单测试数据需满足验证规则**
- **问题**: 测试使用 `password="wrong"`（5 位）触发表单验证，而非 API 登录失败测试
- **根本原因**: 不了解表单的验证规则（最小长度 6 位）
- **规则**: E2E 测试使用的数据必须满足所有前端验证规则
- **实践**:
  - 用户名：≥3 位（示例：`testuser`）
  - 密码：≥6 位（示例：`password123`）
  - 邮箱：符合正则格式
  - 手机号：符合正则格式

### 开发环境规则

**8. Vite 缓存可能导致旧代码运行**
- **问题**: 修复 `IconExchange` 图标后，测试仍然报错，因为 Vite 缓存使用未编译的新代码
- **根本原因**: Vite 的 HMR（热模块替换）缓存未清理
- **规则**: 重大代码修改（特别是核心依赖）后，必须清除缓存并重启服务器
- **实践**:
  ```bash
  pkill -f vite              # 停止开发服务器
  rm -rf node_modules/.vite   # 删除 Vite 缓存
  npm run dev                 # 重启服务器
  ```

基于 Internal Web App 类型的特点，确认以下技术决策：

| 技术决策   | 决策方案                           | 理由                   |
| ---------- | ---------------------------------- | ---------------------- |
| **架构模式**   | SPA（前后端分离）                  | Vue 3 + Sanic 标准架构，用户体验流畅 |
| **浏览器支持** | 现代浏览器（Chrome 90+、Edge 90+、Safari 14+） | 企业内部可控环境，降低兼容复杂度 |
| **SEO**        | 不需要                             | 内部系统，无需搜索引擎索引 |
| **实时功能**   | 定时轮询（非 WebSocket）           | 预警通知每 5 分钟轮询，仪表盘每 30 秒刷新，降低复杂度 |
| **可访问性**   | 基础可访问性（键盘导航、语义化 HTML） | 内部系统，满足基本无障碍需求 |
| **响应式设计** | 是                                 | 支持销售移动访问客户信息 |

**框架选择：**
- **Vue 3**：组合式 API，组件化开发

**架构模式：**
- **SPA 单页应用**：前后端分离，RESTful API 通信

- 首屏加载时间：<3 秒（4G 网络）

CREATE INDEX idx_transfer_from ON transfer_history(from_sales_id, created_at);
CREATE INDEX idx_transfer_to ON transfer_history(to_sales_id, created_at);

-- 性能优化说明
-- 1. 复合索引优先：将常用查询条件组合成复合索引
-- 2. 覆盖索引：查询字段包含在索引中，避免回表
-- 3. 索引维护：定期 ANALYZE TABLE 更新统计信息
-- 4. 查询优化：避免 SELECT *，只查询需要的字段
-- 5. 慢查询监控：记录执行时间>1 秒的查询

**查询优化示例：**

```sql
-- ✅ 好的查询（使用覆盖索引）

FROM customers 
WHERE sales_rep_id = 100 AND tier_level IN ('S', 'A');

-- ❌ 避免的查询（全表扫描）
SELECT * FROM customers WHERE name LIKE '%科技%';

-- ✅ 优化的模糊查询（使用全文索引）
SELECT id, name FROM customers 
WHERE MATCH(name) AGAINST('科技' IN BOOLEAN MODE);
```

### 安全架构

**认证机制：**
- **JWT Token**：无状态认证，过期时间 2 小时
- **Refresh Token**：7 天有效期，支持自动刷新
- **密码加密**：bcrypt 算法，salt rounds=10

**授权机制：**
- **RBAC 模型**：基于角色的访问控制
- **4 级权限**：Admin / Manager / Specialist / Sales
- **数据权限**：销售仅可查看自己客户，运营可查看全部

**数据安全：**
- **敏感信息加密**：联系方式等字段 AES-256 加密存储
- **SQL 注入防护**：参数化查询，ORM 预处理
- **XSS 防护**：前端输入过滤，后端输出编码

**审计日志：**
- **关键操作记录**：增删改、转移、导出、配置变更
- **操作追溯**：操作人、时间、IP、详情
- **日志保留**：至少 12 个月

### 技术债务管理

**已知妥协：**
1. **定时轮询替代 WebSocket**：降低初期复杂度，但预警延迟最高 5 分钟
2. **基础可访问性**：满足基本需求，但不追求 WCAG AA 认证
3. **现代浏览器支持**：不支持 IE11，降低兼容成本

**未来优化机会：**
1. **实时通知**：用户量增长后可升级为 WebSocket
2. **高级可访问性**：根据用户需求提升到 WCAG AA
3. **缓存优化**：高频查询可引入 Redis 缓存
4. **读写分离**：数据量增长后可考虑主从复制

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP 方法：问题解决型 MVP**

聚焦最痛的痛点：结算耗时、数据分散、僵尸账号无法识别

**核心价值验证：**
- 能否将结算时间从 3 天缩短到 2 小时？
- 能否 100% 识别僵尸账号？
- 能否实现客户数据 100% 集中管理？

**资源需求：**
- 前端开发：1 人（Vue 3 + TypeScript）
- 后端开发：1 人（Python Sanic + PostgreSQL）
- UI 设计：0.5 人（兼职，使用 Arco Design 组件库）
- 产品经理：0.5 人（兼职，负责需求确认和验收）
- **总计：3 人，3.5 个月**

---

**核心用户旅程支持：**
1. 运营专员 - 月度结算流程（结算单自动生成）
2. 运营经理 - 客户健康度监控与规则配置
3. 销售团队 - 客户拜访前准备（基础响应式）
4. 运营专员 - 客户转移操作

**Must-Have 功能清单：**

- ✅ 客户详情：完整信息展示、编辑、历史记录
- ✅ 客户新增：手工录入、必填项验证
- ✅ 批量导入：Excel 导入、数据验证、错误处理

**模块 3：客户价值评估**
- ✅ 价值等级显示：S/A/B/C/D 五级展示

- ✅ 结算单生成：选择月份→一键生成（自动关联用量和价格）

### Risk Mitigation Strategy

**技术风险：**

| 风险点             | 风险等级 | 影响         | 缓解策略                             |
| ------------------ | -------- | ------------ | ------------------------------------ |
| **结算单生成性能**   | 中       | MVP 核心价值 | 提前性能测试，优化 SQL 查询，批量处理 |
| **预警推送延迟**     | 低       | 用户体验     | 定时轮询（5 分钟），可接受延迟       |
| **数据迁移复杂性**   | 中       | 上线进度     | 提前演练，编写迁移脚本，预留缓冲时间 |
| **外部接口对接**     | 高       | 第二期依赖   | 提前沟通接口规范，准备 mock 数据     |

**缓解方法：**
- 结算性能：MVP 阶段进行压力测试，确保 1000+ 客户在 1 分钟内完成
- 预警延迟：接受 5 分钟延迟，用户量增长后升级为 WebSocket
- 数据迁移：提前 2 周开始准备迁移脚本，进行 3 轮演练
- 外部接口：第二期开始前 1 个月启动接口对接工作

### 数据迁移计划（Data Migration Plan）

**迁移目标**：将 1320 个客户数据从 Excel 迁移至 PostgreSQL 数据库，数据完整性≥99%

**迁移策略**：三阶段迁移法（第 4-15 周）

**阶段 1：迁移准备**（第 4-6 周，与开发并行）
- 数据清洗：运营专员清洗 Excel（去重、补全）
- 映射规则：字段映射文档、转换规则
- 迁移脚本：Python 迁移脚本、验证脚本
- 环境准备：独立迁移环境、备份机制

**阶段 2：迁移演练**（第 7-9 周，3 轮）
- 第 1 轮（100 条）：验证脚本逻辑
- 第 2 轮（500 条）：验证性能和异常处理
- 第 3 轮（1320 条）：完整演练（含回滚）

**阶段 3：正式迁移**（第 15 周，上线周末）
- 周六 9:00：开始迁移，备份数据
- 周六 10:00：执行迁移脚本（30 分钟）
- 周六 10:30：数据验证（自动 + 人工，抽样 10%）
- 周六 11:30-15:30：问题修复（预留 4 小时缓冲）
- 周日 9:00：系统上线部署
- 周日 14:00：冒烟测试
- 周日 17:00：上线成功确认

**成功标准**：
- ✅ 数据完整性≥99%（错误≤13 条）
- ✅ 关键数据 100% 准确
- ✅ 迁移时间<4 小时
- ✅ 可回滚（失败后 2 小时内恢复）
- ✅ 周一用户可正常使用

**回滚计划**：
- 触发条件：数据错误>50 条，或关键数据错误>5 条
- 回滚步骤：停止系统→恢复备份→验证→通知→分析修复

## Non-Functional Requirements

### NFR 范围说明

**包含的 NFR 类别：**
1. ✅ 性能（Performance）- 用户体验关键，效率提升核心
2. ✅ 安全（Security）- 客户数据保护，权限控制必需
3. ✅ 可靠性（Reliability）- 日常业务依赖，但非 7x24 关键
4. ✅ 可扩展性（Scalability）- 未来增长考虑，但初期压力不大
5. ✅ 集成（Integration）- 第二期需要，MVP 暂不需要

**跳过的 NFR 类别：**
1. ❌ 可访问性（Accessibility）- 内部系统，基础支持即可，不单独定义 NFR

---

### Performance（性能）

**P1: 用户交互响应时间**

| 操作类型         | 响应时间目标 | 测量方法           | 优先级 |
| ---------------- | ------------ | ------------------ | ------ |
| **客户列表加载**   | <1 秒        | 95% 请求           | P0     |
| **客户详情加载**   | <1 秒        | 95% 请求           | P0     |
| **健康度仪表盘**   | <2 秒        | 95% 请求           | P0     |
| **结算单生成**     | <1 分钟      | 1000+ 客户         | P0     |
| **运营报告生成**   | <30 秒       | 周报/月报          | P0     |
| **客户转移执行**   | <30 秒       | 100 个客户         | P0     |
| **等级重算**       | <1 分钟      | 1320 个客户        | P0     |
| **基础搜索**       | <1 秒        | 简单查询           | P0     |
| **多条件筛选**     | <2 秒        | 复杂查询           | P0     |

**P2: 并发用户支持**

- 并发用户数：≥50 用户同时在线
- 峰值性能：并发用户增长到 100 人时，响应时间下降<50%
- 数据库连接池：支持 50+ 并发连接

**P3: 大数据量处理**

- 列表分页：单页最多显示 100 条数据
- 批量导入：支持 1000+ 条/次，处理时间<5 分钟
- 批量导出：支持 1000+ 条/次，处理时间<2 分钟
- 数据量增长：客户数据增长到 10,000 条时，查询性能下降<20%

**P4: 前端性能**

- 首屏加载时间：<3 秒（4G 网络）
- 页面切换时间：<1 秒
- 移动端加载时间：<5 秒（4G 网络）

**性能验证方法：**
- 自动化性能测试（JMeter/k6）
- 关键 API 响应时间监控
- 前端性能监控（Lighthouse）

### Security（安全）

**S1: 认证安全**

- 密码加密：bcrypt 算法，salt rounds=10
- JWT Token：有效期 2 小时，支持自动刷新
- Refresh Token：有效期 7 天
- 登录失败限制：连续 5 次失败后锁定账号 30 分钟
- 密码策略：最少 8 位，包含大小写字母和数字

**S2: 授权安全**

- RBAC 权限模型：4 级权限（Admin/经理/专员/销售）
- 数据权限：销售仅可查看自己负责的客户
- API 权限验证：所有 API 端点必须验证权限
- 会话管理：支持多设备登录，单个用户最多 3 个活跃会话

**S3: 数据安全**

- 传输加密：HTTPS/TLS 1.2+
- 静态加密：敏感字段（联系方式、地址）AES-256 加密存储
- 数据脱敏：日志中不记录敏感信息
- SQL 注入防护：参数化查询，ORM 预处理语句

**S4: 审计日志**

- 关键操作记录：增删改、转移、导出、配置变更、登录登出
- 日志内容：操作人、时间、IP 地址、操作类型、目标对象、详情
- 日志保留：至少 12 个月
- 日志查询：支持按用户、操作类型、时间范围筛选

**S5: XSS 和 CSRF 防护**

- 输入过滤：前端输入验证和过滤
- 输出编码：后端输出 HTML 编码
- CSRF Token：所有状态变更请求必须携带 CSRF Token
- Content Security Policy：限制外部资源加载

**S6: 漏洞管理**

- 依赖扫描：每月扫描 npm/pip 依赖漏洞
- 安全更新：高危漏洞 48 小时内修复
- 渗透测试：上线前进行一次渗透测试

**安全验证方法：**
- 自动化安全扫描（OWASP ZAP）
- 定期渗透测试
- 依赖漏洞扫描（Snyk/GitHub Security）

### Reliability（可靠性）

**R1: 系统可用性**

- 可用性目标：≥99.5%（工作日 9:00-18:00）
- 计划内维护：每月不超过 1 次，提前 48 小时通知
- 非计划停机：每月不超过 4 小时

**R2: 故障恢复**

- 平均故障恢复时间（MTTR）：<2 小时
- 数据恢复点目标（RPO）：<1 小时（最多丢失 1 小时数据）
- 恢复时间目标（RTO）：<4 小时

**R3: 数据备份**

- 备份频率：每日自动备份（凌晨 2:00）
- 备份保留：至少 30 天
- 备份验证：每月进行一次恢复演练
- 备份完整性：100% 可恢复

**R4: 错误处理**

- 用户友好错误信息：不暴露技术细节
- 错误日志：完整记录错误堆栈和上下文
- 错误告警：严重错误（5xx）立即通知运维
- 降级策略：非核心功能失败不影响核心功能

**R5: 数据完整性**

- 数据准确性：≥99.9%（无数据丢失或损坏）
- 事务完整性：所有数据库操作支持事务
- 数据验证：输入数据格式验证、业务规则验证

**可靠性验证方法：**
- 自动化监控（Prometheus + Grafana）
- 定期备份恢复演练
- 故障注入测试（Chaos Engineering）

### Scalability（可扩展性）

**SC1: 用户增长支持**

- 初期容量：50 并发用户
- 中期容量：200 并发用户（12 个月内）
- 长期容量：500 并发用户（3 年内）
- 性能目标：用户增长 10 倍，性能下降<10%

**SC2: 数据增长支持**

- 初期容量：1,320 个客户
- 中期容量：10,000 个客户（12 个月内）
- 长期容量：50,000 个客户（3 年内）
- 性能目标：数据量增长 10 倍，查询性能下降<20%

**SC3: 水平扩展能力**

- 应用层：支持多实例部署，通过负载均衡分发请求
- 数据库：支持读写分离（未来扩展）
- 缓存层：支持 Redis 集群（未来扩展）

**SC4: 功能扩展能力**

- 模块化架构：新功能可独立开发和部署
- API 版本控制：支持/api/v1/, /api/v2/等版本
- 插件机制：支持未来扩展第三方集成

**可扩展性验证方法：**
- 压力测试（模拟 10 倍用户量）
- 数据量模拟测试（导入 10,000+ 测试数据）
- 水平扩展演练（增加/减少实例数）

### Integration（集成）

**I1: 外部接口（第二期）**

- 用量数据采集接口：每日自动获取客户用量数据
- 接口协议：RESTful API 或 Webhook
- 数据格式：JSON
- 认证方式：API Key 或 OAuth 2.0
- 错误处理：重试机制（最多 3 次）、错误告警

**I2: 数据导入导出**

- Excel 导入：支持.xlsx 格式，最大 10MB
- Excel 导出：支持.xlsx 格式，自定义字段
- PDF 导出：标准模板，支持批量生成

**I3: 邮件集成**

- 邮件发送：SMTP 协议
- 邮件模板：支持自定义模板
- 邮件日志：记录发送状态、接收人、时间

**I4: 未来集成预留**

- 财务系统对接（第二期）：与金蝶/用友等财务软件集成
- 单点登录（SSO）：支持 OAuth 2.1/OIDC（未来扩展）
- API 开放：支持第三方系统集成（未来扩展）

**集成验证方法：**
- 接口联调测试
- 数据一致性验证
- 错误场景测试（网络中断、数据格式错误等）

### NFR 权衡与决策

**已知的权衡决策：**

| 决策点               | 决策结果     | 理由                           | 未来优化               |
| -------------------- | ------------ | ------------------------------ | ---------------------- |
| **实时通知**           | 定时轮询     | 降低初期复杂度                 | 用户量增长后升级 WebSocket |
| **数据库读写分离**     | 暂不支持     | 初期数据量不大，单库足够       | 数据量>10,000 时考虑   |
| **Redis 缓存**         | 暂不需要     | 降低架构复杂度                 | 高频查询场景增加缓存   |
| **高级可访问性**       | 基础支持     | 内部系统，无特殊需求           | 根据用户需求提升       |
| **多租户架构**         | 不支持       | 内部系统，单租户足够           | SaaS 化时重新设计      |

### NFR 监控与持续改进

**监控指标：**

- 性能监控：API 响应时间、前端加载时间、数据库查询时间
- 安全监控：登录失败次数、异常访问检测、漏洞扫描结果
- 可靠性监控：系统可用性、错误率、备份成功率
- 容量监控：并发用户数、数据量增长、存储使用率

**改进机制：**

- 每月审查：NFR 指标审查，识别改进机会
- 用户反馈：收集用户关于性能、可用性的反馈
- 技术债务：定期评估和偿还技术债务
- 架构演进：根据业务增长调整架构
